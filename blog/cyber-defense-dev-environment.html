Building a Cyber Defense Development Environment in a Virtual Machine

Norris Cornell · Blue Team · Lab Setup · Python / Network Security

In today's rapidly evolving cyber threat landscape, having a robust and flexible cybersecurity development environment is crucial for professionals aiming to enhance their defensive capabilities. This guide walks through setting up a powerful, portable development environment inside an Ubuntu XFCE virtual machine — designed to support AI-driven cybersecurity experiments and sophisticated network defense tools. Whether you're just getting started or expanding your skills, this gives you a reproducible blue-team lab you can take anywhere.

STEP 01 Set Up the Base Environment

Download and install VirtualBox. Then grab an Ubuntu XFCE ISO image — it gives you a lightweight Linux desktop without the overhead of a full GNOME or KDE environment. Create a new virtual machine with the following baseline settings:

SETTING VALUE NOTE
RAM 2 GB minimum 4 GB recommended if running ML models later
Disk Space 20 GB Dynamically allocated is fine
Network (default) NAT Internet access through host; safest default
Network (alternate) Bridged Adapter Direct access to your home network — needed for ARP scanning

Mount the ISO, run through the installer, then update your packages and install the Python toolchain:

TERMINAL 
$ sudo apt update
$ sudo apt install python3 python3-pip python3-venv
STEP 02 Create an Isolated Python Environment

Use Python's built-in venv module to isolate your development environment. This ensures installed libraries don't interfere with system-wide packages — and it makes your lab reproducible and portable.

TERMINAL 
$ python3 -m venv ai-env
$ source ai-env/bin/activate
(ai-env) $ pip install tensorflow torch gymnasium stable-baselines3 jupyter

Always activate your virtual environment before working on a project. It keeps dependencies scoped, makes collaboration cleaner, and prevents the "it worked on my machine" problem.

STEP 03 Build Your First Cyber Defense Agent

Start simple: a ping-based host discovery agent. This establishes the basic pattern — iterate across a subnet, check for responsive hosts, log results — that everything more sophisticated builds on.

TERMINAL 
(ai-env) $ pip install ping3
◈ PYTHON · ping_discovery.py 
# Basic ICMP host discovery across a /24 subnet
from ping3 import ping

hosts = ["192.168.1." + str(i) for i in range(1, 255)]
active_hosts = []

for host in hosts:
    response = ping(host, timeout=1)
    if response:
        print(f"Host {host} is active.")
        active_hosts.append(host)

print(f"Active hosts: {active_hosts}")
STEP 04 Upgrade to ARP Scanning with Scapy

Move beyond ICMP. ARP scanning is more reliable for local network discovery — it works even when hosts are configured to block ping. Scapy gives you the low-level packet crafting capability to do this properly.

TERMINAL 
$ python3 -m venv scanner-env
$ source scanner-env/bin/activate
(scanner-env) $ pip install scapy netifaces
◈ PYTHON · arp_host_scanner.py 
# ARP-based local network discovery using Scapy
from scapy.all import ARP, Ether, srp
import netifaces

def arp_scan(network):
    arp    = ARP(pdst=network)
    ether  = Ether(dst="ff:ff:ff:ff:ff:ff")
    packet = ether / arp

    result  = srp(packet, timeout=3, verbose=0)[0]
    devices = []

    for sent, received in result:
        devices.append({
            'ip':  received.psrc,
            'mac': received.hwsrc
        })

    return devices

ARP operates at Layer 2 — it reaches devices that don't respond to ICMP, and it returns MAC addresses alongside IPs, giving you vendor identification capability when cross-referenced against an OUI database.

STEP 05 Resolve Permissions and Virtual Environment Access

ARP scanning requires raw socket access — which means root privileges. The trick is to invoke the virtual environment's Python interpreter directly via sudo, rather than relying on the system Python. This preserves your venv libraries while running with elevated permissions.

TERMINAL 
# Run the scanner with root privileges using the venv interpreter
$ sudo ./scanner-env/bin/python arp_host_scanner.py

Running sudo python3 directly would use the system Python, bypassing your virtual environment. Always point sudo at the venv interpreter explicitly when elevated access is required.

STEP 06 Enhance with Hostname Resolution

IP addresses and MAC addresses tell you where something is. Hostnames tell you what it is. Add reverse DNS lookups to make your scan output immediately interpretable — no manual cross-referencing required.

◈ PYTHON · hostname_resolution.py 
import socket

def resolve_hostname(ip):
    try:
        # Attempt reverse DNS lookup
        return socket.gethostbyaddr(ip)[0]
    except socket.herror:
        # No PTR record — return Unknown rather than crashing
        return 'Unknown'

Not all devices will have PTR records — embedded systems, PLCs, and IoT devices often don't. The try/except ensures the scanner keeps running when lookups fail rather than terminating mid-scan.

STEP 07 Validate and Log Results

Scan results are only useful if you can compare them across time. Log every scan to a timestamped file — this gives you a baseline to diff against, making new or unauthorized devices immediately visible in future runs.

◈ PYTHON · scan_logger.py 
import datetime
import os

# Ensure logs directory exists
os.makedirs("logs", exist_ok=True)

# Generate a timestamped filename for this scan
log_filename = datetime.datetime.now().strftime(
    "logs/scan_%Y%m%d_%H%M%S.txt"
)

with open(log_filename, 'w') as log_file:
    for device in devices:
        hostname = resolve_hostname(device['ip'])
        log_file.write(
            f"IP: {device['ip']}  MAC: {device['mac']}  Hostname: {hostname}\n"
        )

print(f"Scan logged to: {log_filename}")

With timestamped logs in place, you can diff any two scan files to see exactly what changed — which devices appeared, which disappeared, and which changed their MAC address (a classic indicator of a spoofing attempt or a rogue device replacement).
