Logic Follows Lies: How PLCs and RTUs Fail Under Adversarial Conditions

If inputs lie, logic will follow.
  • Industrial control systems don't reason.
  • They don't question.
  • They don't validate truth.
  • They evaluate.

A PLC or RTU accepts whatever value reaches its input buffers and applies deterministic logic to it with absolute confidence. And when those upstream signals are manipulated — whether sensor readings, timing references, or permissive bits — the controller's logic becomes the attack surface.

In Part 2 of this series (building on the BSides Delaware presentation), we explored how industrial systems trust external signals that were never designed for contested environments: GNSS timing, power system telemetry, and navigation signals with no integrity controls. Now, in Part 3, we move downstream — into the controller itself — to expose why a single falsified input can trigger state transitions, bypass interlocks, destabilize PID loops, or issue dangerous physical commands.

Why PLC Logic Is Fragile When Inputs Are Manipulated

PLCs operate on a small set of truths:

// PLC EVALUATION MODEL IF rung condition is met → evaluate next instruction IF permissive is true → enable the sequence IF timer expires → advance to next state IF value crosses threshold → activate output // NOTICE WHAT'S MISSING: ✗ "Does this value make physical sense?" ✗ "Is this sensor behaving consistently with others?" ✗ "Did this input change too quickly to be real?"

There is no truth verification inside the controller. A manipulated input becomes a manipulated state — and the controller reacts with deterministic certainty, even if the reaction is hazardous.

Three Failure Modes When Inputs Are Manipulated

// FAILURE MODE 01

Forced State Transitions

A spoofed sensor value crosses a threshold that the physical system never reached. The state machine changes — and no alarms trigger, because every downstream rung evaluates normally.

  • Falsified temperature drops 180°F → 85°F: heater activates, overheating the process
  • Manipulated tank sensor reads "low level": pump starts, causing cavitation or dry-run damage
  • Falsified pressure shows "safe": safety interlock bypassed, dangerous sequence initiates

One manipulated value changes everything downstream.

// FAILURE MODE 02

Logic Pollution

The most underrated ICS attack class. An attacker changes a single upstream value — a bit, a counter, a mode flag, a timer preset, a permissive — and that corrupted value propagates through:

  • Latching coils and safety rails
  • Step sequences and PID controllers
  • Startup interlocks and permissive chains

This is not a single control failure — this is logical corruption that propagates like a toxin.

// FAILURE MODE 03

Timing Desynchronization

Timing attacks are the silent killers of industrial logic. Most ICS environments rely on GNSS or NTP for synchrophasor alignment, breaker coordination, relay comparison logic, event sequencing, and protection logic ordering.

When an attacker spoofs or shifts timing — no malware, no firmware modification, no protocol injection needed. The system is simply confused.

  • Relays misfire
  • Interlocks expire too early or too late
  • Controllers advance steps out of order

The system still "runs" — but not safely.

Real-World Attack Scenarios

// SCENARIO 01

RTU Desynchronization → Breaker Trips at the Wrong Time

Grid protection relays depend on tightly synchronized phasors. Spoof GNSS timing and you poison the foundation of grid coordination. A few microseconds of timing drift can cascade into a grid event.

  • False breaker openings or failure to trip when needed
  • Out-of-phase switching across substations
  • Incorrect relay comparisons and protection misalignment
// SCENARIO 02

Forced Permissive → Safety Interlock Bypassed

Safety sequences rely on permissives: tank must be full, valve must be closed, pressure must be stable. If an attacker flips one bit, the PLC sees a safe condition that does not exist — and proceeds with complete confidence.

  • Startup, shutdown, pressurization, chemical dosing
  • Heating cycles and mechanical actuation
  • All initiated against a physical state the system cannot see
// SCENARIO 03

PID Loop Instability Through Sensor Drift

PID loops trust input values completely. If an attacker gradually spoofs a sensor, the loop tries to correct a problem that isn't there — outputs oscillate, actuators hunt, processes destabilize, equipment is stressed or damaged.

The most dangerous pattern: operators interpret the instability as a mechanical issue, not an adversarial one. The attack hides behind normal-looking system behavior.

Why Defenders Miss These Attack Paths

Most ICS security programs focus on the right things — but not sufficient things:

// WHAT DEFENDERS TYPICALLY COVER
  • Patch management
  • Firmware integrity
  • Secure configuration
  • Network segmentation
  • Logging & monitoring
  • Protocol inspection
  • Firewalls & allowlists

These frameworks treat the signal layer as "trusted infrastructure, not an attack surface." Defenders focus on the network, controller, and HMI layers — but not the most manipulable layer of all: the inputs themselves.

Why This Problem Is So Fundamental

Because the control system is not the victim. Reality is the victim.

When attackers manipulate inputs:

// THE CORE INSIGHT

The PLC is not hacked. The RTU is not compromised.
The HMI is not infected. The historian is not altered.

The process is manipulated through logic behaving exactly as designed.

This is what makes signal-layer attacks so effective — and so hard to defend against. Because the attacker doesn't need to exploit the controller. They just need to lie convincingly.

Physics-Based Detection & Signal Verification

  • Physics-Based Anomaly Detection

  • Timing Drift Detection

Every industrial cyberattack begins with the same principle: "If you can control what the system sees, you can control what it does."

PLCs and RTUs don't validate truth — they only evaluate inputs. This makes the signal layer one of the easiest, most cost-effective targets for adversaries, yet one of the least defended.

In an era where operational environments are increasingly contested, defenders must stop treating sensors, permissives, and external timing as trusted infrastructure.

Because logic follows lies. And in ICS environments, lies can break the physical world.
